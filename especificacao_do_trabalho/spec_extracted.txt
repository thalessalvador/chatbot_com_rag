4.2  Chunking  (segmentação)  e  preparação  
Implementar  uma  estratégia  de  chunking  e  justificar  decisões.  
Requisitos  mínimos  
●  Cada  chunk  deve  ter:  
 ○  chunk_id único  
○  doc_id de  origem  ○  texto  do  chunk  ○  metadados  herdados  do  documento  (pelo  menos  titulo e  fonte)

4.3  Recuperação  ( retriever )  
Implementar  um  retriever  denso :  
●  embeddings  +  índice  vetorial  (FAISS/Chroma/pgvector/etc.)  ●  retorno  de  top-k  chunks  com  scores  e  metadados  
Requisitos  mínimos  
●  Permitir  configurar  k (pelo  menos  3,  5  e  10).  ●  Expor  (ao  menos  no  log)  quais  chunks  foram  recuperados  e  seus  identificadores.  
4.4  Pipeline  RAG  com  citações  –  obrigatório  
O  chatbot  deve  executar  o  pipeline:  
1.  receber  pergunta  do  usuário  2.  recuperar  top-k  chunks  do  corpus  ( retrieval )  3.  montar  prompt  com  contexto  recuperado  4.  gerar  resposta  apenas  baseada  nas  fontes  5.  retornar  resposta  com  citações  referenciando  chunks/documentos  
Requisitos  mínimos  
●  Grounding:  o  prompt  deve  instruir  claramente  para  usar  somente  o  contexto  
recuperado.
 ●  Citações:  a  resposta  deve  incluir  referências  a  chunks  (ex.:  [doc_12#chunk_03]).  ●  Recusa  adequada:  se  não  houver  evidência  suficiente,  responder  “não  encontrei  na  
base”
 
(ou
 
equivalente).

5)  Avaliação  obrigatória  
5.1  Golden  set  
Construir  um  conjunto  de  aproximadamente  20  perguntas  ( golden  set )  cobrindo:  
●  perguntas  factuais  diretas  ●  perguntas  que  exigem  combinar  2+  trechos  ●  perguntas  fora  do  corpus  (para  testar  recusa)  
Cada  pergunta  deve  ter:  
●  a  própria  pergunta  ●  indicação  do(s)  chunk(s)  “relevantes  esperados”  ou  critério  claro  do  que  conta  como  
evidência
 
5.2  Métricas  mínimas  
1.  Recall@k  do  retriever  denso  (k=3,5,10)  ○  O  chunk  esperado  aparece  no  top-k?  2.  Avaliação  qualitativa  (rubrica)  em  pelo  menos  15  perguntas  do  golden  set :  ●  Groundedness:  resposta  está  suportada  pelos  trechos?  ●  Correção:  está  correta  dado  o  corpus?  ●  Citações:  cita  trechos  adequados  e  coerentes?  ●  Alucinação:  inventou  algo  fora  do  corpus?  ●  Recusa:  quando  não  há  evidência,  recusou  corretamente?  
Ferramentas  automáticas  (ex.:  RAGAS)  são  opcionais.  A  rubrica  manual  é  
suficiente
 
se
 
bem
 
feita.

6)  Trilhas  de  melhoria  (escolher  uma)  
Além  do  baseline,  cada  grupo  deve  escolher  e  implementar  uma  trilha  dentre  as  listadas  
abaixo.
 
Regras  gerais  para  qualquer  trilha  
●  Deve  haver  comparação  com  o  baseline .  ●  Deve  haver  evidência  de  ganho  (métricas  e/ou  rubrica  com  exemplos).  ●  Parâmetros  e  scripts  devem  estar  versionados  no  repositório.  
Trilha  A:  Recuperação  Híbrida  ( Sparse  +  Dense )  
Objetivo  
Aumentar  o  recall  e  robustez  do  retrieval  combinando  recuperação  esparsa  e  densa .  
O  que  implementar  
1.  Implementar  um  retriever  esparso  (BM25  ou  TF-IDF).  2.  Implementar  um  modo  híbrido  que  combina  sparse+dense  (ex.:  RRF,  soma  
ponderada
 
ou
 
outra
 
fusão).
 3.  Expor  na  apresentação  os  3  modos:  dense (baseline),  sparse,  hybrid.  
Como  avaliar  
●  Calcular  Recall@k  (k=3,5,10)  para  os  três  modos.  ●  Apresentar  análise  curta  de  trade-offs  (quando  híbrido  melhora/piora).  
Critérios  mínimos  
●  Deduplicação  de  chunks  na  fusão.  ●  Descrever  parâmetros  do  sparse  e  do  dense.

Trilha  B:  Reranking  (2ª  etapa  de  ranking)  
Objetivo  
Melhorar  precisão:  fazer  com  que  os  top-k  chunks  usados  pelo  RAG  sejam  mais  relevantes.  
O  que  implementar  
1.  Recuperar  top-N  com  o  retriever  baseline  (ex.:  N=20).  2.  Aplicar  um  reranker  para  reordenar  e  selecionar  top-k  final  (ex.:  k=5–10).  ○  Pode  ser  cross-encoder,  LLM  reranker  com  prompt  estruturado,  utilização  de  
modelo
 
leve,
 
entre
 
outros.
 
Como  avaliar  
●  Comparar  baseline  vs  reranking  usando:  ○  rubrica  qualitativa  (relevância/groundedness/citações)  no  golden  set .  ○  opcionalmente  Recall@k  e/ou  nDCG  ●  Reportar  impacto  em  tempo/latência.  
 
Critérios  mínimos  
●  Definir  N  e  k  e  justificar.  ●  Demonstrar  que  a  ordem/seleção  muda  e  melhora  casos.

coerência
 
e
 
resolver
 
referências.

1.  Persistir  o  histórico  em  unidades  recuperáveis  (turnos/blocos)  com  metadados

 3.  Em  cada  nova  pergunta:  ○  recuperar  top-M  trechos  do  histórico,  ○  combinar  com  o  contexto  do  corpus,  ○  aplicar  política  de  prioridade  (memória  vs  corpus).  4.  Transparência:  mostrar  quais  turnos/trechos  foram  usados.  
Como  avaliar  
●  Criar  10–15  cenários  conversacionais  e  medir:  ○  Context  Recall  (recuperou  o  trecho  certo?)  ○  Context  Precision  (trouxe  irrelevâncias?)  ●  Rubrica  qualitativa:  coerência  e  ausência  de  “memórias  inventadas”.  
Critérios  mínimos  
●  Não  vale  “jogar  todo  o  histórico  no  prompt”.  ●  Política  explícita  de  uso  e  prioridade.
